Intro to Selenium IDE

  1. A few facts about Selenium

    1. A suite of tools used for web page testing

    2. Available via OpenQA, home for many open-source QA tools

    3. Runs in several browsers: Internet Explorer, Mozilla, and Firefox on Windows, Macintosh, and Linux platforms

    4. Named for the antidote for mercury poisoning, as a "jab" at Mercury Interactive, makers of the well-known WinRunner tool for testing web pages and applications

  2. Differences between Selenium and WinRunner/SilkTest/QTP/TestComplete

    1. Selenium is free! The others range from several hundred to several thousand in price.

    2. Selenium is limited to web page testing. The others can also test Windows applications.

    3. Selenium can be run on a Mac OS or Linux system. The others run only on Windows platforms.

    4. Selenium can be used for browser compatibility testing. The others are very limited in this regard.

  3. Selenium projects

    1. Selenium IDE (Integrated Development Environment)

      1. Implemented as a Firefox extension

      2. Provides record/replay capabilities

      3. Allows user scripting only in Selenese language

      4. The focus of this course

    2. Selenium RC (Remote Control)

      1. Allows web tests to be written in a variety of high-level programming languages: Java, Perl, PHP, Python, Ruby, JavaScript, .NET.

      2. The focus of the next course in Selenium at De Anza

    3. Selenium Core

      1. Must be installed on same server as SUT

      2. Allows user scripting in Selenese language

      3. Covered lightly in this course

    4. Selenium on Rails

      1. Used to test Ruby on Rails apps with Selenium Core

      2. Provides additional formats besides HTML for scripts

      3. Not covered in either this course or the next one

  5. Selenium IDE Example: Link-Checking

    1. Recording

      1. Point Firefox browser at SUT, De Anza A-Z Directory.

      2. Select File=>New Test Case.

      3. Click on the red Record button to start recording (it is already on by default when Selenium IDE is first brought up).

      4. Click on browser window to give it focus.

      5. Click on letter T from alphabetical horizontal navigation bar.

      6. Select "Technical Writing" link, and wait for new page to load.

      7. Bring up context menu.

        1. Right-click for right-handed Windows users.

        2. Left-click for left-handed Windows users.

        3. CTRL-click for Mac users.

      8. Select Verify Title Technical Writing from context menu.

      9. Select browser's Back button.

      10. Repeat the last four steps for the "Transcripts" link.

      11. Repeat the last four steps for the "Transfer Planning" link.

      12. Click the red Record button to stop recording.

        At this point, all of the user actions except the clicks on the Back button have recorded. To take care of these clicks, we will add calls to the goBackAndWait function.

    2. Inserting Selenese function calls via Table view

      1. Select Table tab of Selenium IDE.

      2. Select second clickAndWait.

      3. Bring up the context menu of IDE and select Insert New Command.

      4. Type "g" into Command field.

      5. Select goBackAndWait from command menu.

      6. Repeat last three steps for third clickAndWait call.

    3. Inserting comments via Table view

      1. From Table mode, select the first clickAndWait call.

      2. Bring up the context menu of IDE and select Insert New Comment.

      3. Enter "Check Technical Writing link...."

      4. Repeat the last three steps for the remaining clickAndWait calls.

    4. Replaying a test

      1. Select File=>Save As to save the test.

      2. Drag the speed slider to Slow.

      3. Select the green right-pointing arrow ("Play from the beginning or start point").

    5. Analyzing the results

      1. From Table view, observe the green vs. red shading on assertions.

      2. From Table or Source view, select the Log tab and look for errors (bold red font).

    6. Entering/modifying Selenese commands via the Source tab.

      1. Select the Source tab and scroll down to the second verifyTitle call.

      2. Eliminate the trailing "s" in "Transcripts."

      3. Select File=>Save.

        This step is critical; otherwise, the changes will be thrown away. This seems to be a bug with changes made via the Source tab, as changes made via the Table tab do not have to be saved before replaying a test.

      4. Replay the test again. Analyze the results. This time, the Table view should display red shading on the second call to verifyTitle and the Source view's Log should contain a bold red font error message.
